### 1.3 위반 판정 로직: Rule-based vs LLM-based

#### 현재 (Rule-based):
```python
# 간단한 키워드 매칭만 함
if "판매" in prompt and "금지" in article:
    violation = True
```

**문제**: 미묘한 위반 사례 놓칠 수 있음

#### 개선 (Hybrid):
```python
# 1차: Rule-based (빠른 필터링)
if explicit_violation_signal(prompt):
    return True

# 2차: LLM-based (정확한 판정)
# Claude API로 프롬프트와 조항을 비교
verdict = claude_api.assess_violation(prompt, article)
```

**이점**:
- Rule-based: 빠르고 확실한 위반 탐지
- LLM-based: 미묘한 위반까지 정확하게 판정

**추천**: **Hybrid (Rule-based + LLM)**

**비용 고려**:
- Rule-based만: 무료 (단, 정확도 낮음)
- LLM-based만: API 비용 증가
- Hybrid: 중간 비용, 최고 정확도 ✓

## 📈 3. 기대 효과

### 3.1 정확도 개선
```
현재 (문제점):
  "분석" 키워드 → laws=[] (모든 법령 제외)
  정확도: ~50% (추정)

개선 후 (Phase A-E 완료):
  하이브리드 검색 + LLM 판정
  정확도: ~95% (목표)
```

### 3.2 처리 속도
```
현재: ~10ms/케이스 (빠름)

개선 후:
  - 캐싱 추가: ~10ms/케이스 (유지)
  - LLM 호출: +100-300ms/케이스 (Rule-based만 적용 시 회피)

최적화: Rule-based로 명확한 위반만 먼저 필터링
        → LLM은 모호한 경우만 호출 (비용 절감)
```

### 3.3 신뢰도
```
현재: Rule-based, 단순 키워드 → 신뢰도 낮음
개선: Rule-based + LLM + 상세 로깅 → 신뢰도 높음
```
