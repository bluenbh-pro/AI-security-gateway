# AI 게이트웨이: 임직원 프롬프트 유형 분석

## 📋 개요
실제 회사 업무 환경에서 임직원들이 외부 LLM(ChatGPT, Claude 등)에 입력할 가능성이 높은 프롬프트 유형들을 정의합니다.

---

## 🔹 Type 1: 파일 첨부 + 간단 요청 (File-Based with Simple Request)

**설명**: 회사 문서/파일을 첨부하고 간단한 작업을 요청하는 유형

**사례:**
```
파일 첨부: 2026_분기별_마케팅_전략.pdf (극비 문서)
프롬프트: "이 문서를 영어로 요약해줄 수 있어?"
```

**민감 정보 패턴:**
- 극비 / 대외비 문서
- 마케팅 전략, 신제품 계획
- 재무 현황, 경영진 지시사항

**위반 가능성:**
- 영업비밀 유출 (영업비밀법)
- 기밀 정보 외부 노출

**위험도**: 🔴 **높음**

---

## 🔹 Type 2: 텍스트 임베딩 + 데이터 분석 (Embedded Data + Analysis)

**설명**: 민감한 데이터를 프롬프트에 직접 포함시키고 분석을 요청

**사례:**
```
프롬프트: 
"우리 고객의 신용점수 데이터는 다음과 같아:
- 고객ID: C-12345, 신용점수: 850, 월소득: 500만원, 직업: 은행원
- 고객ID: C-12346, 신용점수: 720, 월소득: 300만원, 직업: 회사원
이 데이터에 기반해 대출 승인 알고리즘을 개선해줄 수 있어?"
```

**민감 정보 패턴:**
- 신용정보 (신용점수, 대출 기록)
- 개인정보 (이름, 주민번호, 연락처)
- 소득, 직업, 가족 정보
- 거래 기록, 여신 정보

**위반 가능성:**
- 신용정보법 위반 (제3자 제공 금지)
- 개인정보보호법 위반 (무단 처리)
- 금융감독규정 위반

**위험도**: 🔴 **매우 높음**

---

## 🔹 Type 3: 정책/규정 문서 공유 (Policy/Guideline Sharing)

**설명**: 회사의 내부 정책이나 규정을 외부 LLM과 공유

**사례:**
```
프롬프트:
"우리 회사의 신용정보 접근 통제 정책은 다음과 같아:
1. 임원급은 무제한 접근
2. 과장급은 실명 및 신용점수만 접근
3. 일반직은 접근 금지

이 정책이 국제 표준(ISO, GDPR)에 맞는지 피드백 줄 수 있어?"
```

**민감 정보 패턴:**
- 회사 보안 정책
- 접근 통제 규칙 (운영 절차)
- 감시/모니터링 정책
- 내부 승인 프로세스

**위반 가능성:**
- 영업비밀 유출 (운영 절차)
- 정보보안 방침 노출
- 시스템 취약점 노출 우려

**위험도**: 🟠 **중간-높음**

---

## 🔹 Type 4: 고객/거래 데이터 분석 (Customer/Transaction Data Analysis)

**설명**: 고객 거래 데이터를 첨부하여 패턴 분석 요청

**사례:**
```
파일 첨부: 고객_거래기록_2026_Q1.csv (개인정보 + 신용정보 포함)
프롬프트: "이 고객들의 거래 패턴을 분석해서 이상거래를 탐지해줄 수 있어?
또한 이들을 고위험/저위험 그룹으로 분류해줄 수 있어?"
```

**민감 정보 패턴:**
- 거래 기록 (날짜, 금액, 상대방)
- 고객 정보 (이름, 계좌)
- 신용 정보
- 거래 패턴 (이상거래 단서)

**위반 가능성:**
- 개인정보보호법 위반 (개인 거래 내역)
- 신용정보법 위반
- 금융감독규정 위반
- 전자금융거래법 위반

**위험도**: 🔴 **매우 높음**

---

## 🔹 Type 5: 내부 프로세스 개선 (Internal Process Improvement)

**설명**: 회사의 내부 프로세스를 설명하고 개선안 요청

**사례:**
```
프롬프트:
"우리 신용승인 프로세스는:
1. 신청 접수 → 신용조회
2. 신용평점 계산 (우리만의 AI 알고리즘)
3. 리스크 점수링
4. 최종 승인 판정

이 프로세스를 더 효율화할 수 있는 방법이 있을까?"
```

**민감 정보 패턴:**
- 내부 업무 절차
- 승인 기준 및 가중치
- AI 모델 로직
- 리스크 평가 방식

**위반 가능성:**
- 영업비밀 유출
- 승인 기준 외부 노출
- 시스템 설계 노출

**위험도**: 🟠 **중간-높음**

---

## 🔹 Type 6: 컴플라이언스 문서 작성 (Compliance Document Drafting)

**설명**: 회사의 개인정보보호, 신용정보 정책 등을 작성하기 위해 현황 정보 공유

**사례:**
```
프롬프트:
"우리 회사는:
- 고객 2,000만 명의 개인정보 보유
- 신용정보 데이터는 AWS 암호화 저장소에 관리
- 접근자는 임원 10명, 과장 50명

이 현황을 기반으로 개인정보보호 정책안을 작성해줄 수 있어?"
```

**민감 정보 패턴:**
- 고객 수 규모 (경영 지표)
- 데이터 저장 위치 및 방식
- 접근 권한자 규모
- 데이터 보유 범위

**위반 가능성:**
- 경영 정보 유출
- 시스템 인프라 정보 노출
- 조직 구조/인원 정보 노출

**위험도**: 🟠 **중간**

---

## 🔹 Type 7: 코드 검토 (Code Review Request)

**설명**: 회사의 내부 코드를 첨부하여 보안/성능 검토 요청

**사례:**
```
파일 첨부: auth_module.py (내부 인증 시스템 코드)
프롬프트: "이 우리 인증 모듈의 보안 취약점을 찾아줄 수 있어?
특히 토큰 검증 부분에서 우려되는 점이 있어?"
```

**민감 정보 패턴:**
- 소스 코드 (시스템 설계 노출)
- 인증 로직 (보안 취약점 단서)
- API 설계
- 데이터베이스 스키마

**위반 가능성:**
- 영업비밀 유출 (소스 코드)
- 시스템 취약점 노출
- 인증 메커니즘 노출

**위험도**: 🔴 **높음**

---

## 🔹 Type 8: 개인정보 샘플 데이터 생성 (Synthetic Data Generation)

**설명**: 테스트/개발 목적으로 개인정보 유사 샘플 데이터 요청

**사례:**
```
프롬프트:
"우리 고객 데이터 구조는:
- 이름, 주민번호, 전화번호, 이메일, 직업, 월소득, 신용점수

이와 유사한 가짜 고객 데이터 1,000개를 생성해줄 수 있어?
(실제 고객 정보처럼 보이는 데이터)"
```

**민감 정보 패턴:**
- 개인정보 필드 구조 노출
- 실제와 유사한 데이터 패턴

**위반 가능성:**
- 개인정보 구조 정보 노출
- 개인정보 유사 데이터 외부 생성
- 사기 목적에 악용 우려

**위험도**: 🟠 **중간**

---

**사례:**
```

프롬프트: "이 문서를 영어로 번역해줄 수 있어?"
```

**민감 정보 패턴:**
- 극비 / 대외비 문서 그 자체

**위반 가능성:**
- 영업비밀 유출
- 기밀 정보 외부 노출
- 번역 과정에서 제3자 접근 가능

**위험도**: 🔴 **높음**

---

## 🔹 Type 10: 데이터 정제/변환 (Data Cleaning/Transformation)

**설명**: 고객/거래 데이터를 정제하거나 형식 변환 요청

**사례:**
```
파일 첨부: 고객_더티데이터_Q1.csv (중복 제거 필요)
프롬프트: "이 데이터에서 중복된 고객을 제거해줄 수 있어?
그리고 이상치를 찾아줄 수 있어?"
```

**민감 정보 패턴:**
- 고객 정보 (이름, 연락처, 주소)
- 거래 정보
- 신용 정보

**위반 가능성:**
- 개인정보보호법 위반
- 신용정보법 위반
- 데이터 기본법 위반

**위험도**: 🔴 **높음**

## 🔹 Type 11: 자금 거래 관련 조회 (Financial Transaction Inquiry)

**설명**: 구체적인 자금 거래나 계좌 정보 관련 질문

**사례:**
```
프롬프트:
"우리 회사 계좌 (예금주: ABC은행, 계좌번호: 123-456-789)에
어제 5,000만원이 입금되었는데,
이 자금이 어디서 온 건지 추적해줄 수 있어?
거래처 정보도 있는데, 리스크 평가를 해줄 수 있어?"
```

**민감 정보 패턴:**
- 회사 계좌 정보
- 자금 흐름
- 거래처 정보
- 자금 규모

**위반 가능성:**
- 금융 정보 외부 노출
- 자금세탁 의도 의심 (감시)
- 거래 기밀 유출

**위험도**: 🔴 **매우 높음**

---

## 🔹 Type 12: 임원/고위험 직원 정보 (Executive/High-Risk Employee Info)

**설명**: 임원이나 특정 직원의 개인정보/거래 정보 관련 질문

**사례:**
```
프롬프트:
"CEO 김철수(주민번호: XXXXXX-XXXXXX)는 지난 3개월간
신용카드로 해외 거래를 많이 했어.
이 패턴이 정상인지 분석해줄 수 있어?
혹시 신용사기 피해 가능성이 있을까?"
```

**민감 정보 패턴:**
- 임원 개인정보
- 거래 기록
- 개인 신용 정보

**위반 가능성:**
- 개인정보보호법 위반 (고위험자 정보)
- 신용정보법 위반
- 사생활 침해

**위험도**: 🔴 **매우 높음**

## 🔹 Type 13: 규정 회피 질문 (Compliance Evasion Query)

**설명**: 회사의 규정/법령을 어떻게 우회할 수 있는지 묻는 경우

**사례:**
```
프롬프트:
"우리 회사 정책상 고객 신용정보는 제3자에 제공할 수 없어.
하지만 마케팅팀이 고객 신용점수 데이터가 필요해.

법적으로 문제없이 고객 데이터를 
마케팅 회사에 제공할 수 있는 방법이 있을까?"
```

**민감 정보 패턴:**
- 규정 회피 의도
- 법령 해석 요청
- 신용정보 무단 제공 의도

**위반 가능성:**
- 신용정보법 위반 (의도적)
- 개인정보보호법 위반 (의도적)
- 금융감독규정 위반

**위험도**: 🔴 **극도로 높음** (의도적 위반 시도)

---

## 🔹 Type 14: 시스템 침입/보안 질문 (System Intrusion Query)

**설명**: 회사 시스템에 접근하는 방법이나 보안 우회 방법 질문

**사례:**
```
프롬프트:
"우리 회사의 신용정보 시스템은
현재 HTTPS와 IP 화이트리스트로 보호돼.

이 보안을 우회해서 데이터에 접근할 수 있는 방법이 있을까?
(보안 테스트 목적이야)"
```

**민감 정보 패턴:**
- 시스템 보안 아키텍처
- 접근 통제 메커니즘
- 보안 우회 시도

**위반 가능성:**
- 정보통신망 이용촉진 및 정보보호 등에 관한 법률 위반
- 해킹 의도
- 시스템 보안 위협

**위험도**: 🔴 **극도로 높음** (악의적 시도)

---

## 🔹 Type 15: 일반적인 업무 질문 (General Work Query)

**설명**: 민감 정보 없이 순수 업무 관련 질문

**사례:**
```
프롬프트:
"신용평가 모델을 개발하려고 하는데,
일반적인 머신러닝 접근법은 무엇이 있을까?
결정트리, 로지스틱 회귀, 신경망 중 
어떤 방법이 좋을까?"
```

**민감 정보 패턴:**
- 없음 (일반적인 기술 질문)

**위반 가능성:**
- 없음

**위험도**: 🟢 **매우 낮음** (허용)
