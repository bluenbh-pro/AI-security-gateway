# 5개 Agent 상세 검토 분석

**기준**: 원래 프로젝트 목표인 "금융권 LLM 보안 게이트웨이" 역할 충실성

---

## 📋 프로젝트 목표 Node 매핑

```
[Node 1] Prompt Injection 검사          ← Agent 5 ✅
[Node 2] 민감정보 / 기밀도 분류        ← Agent 1 ✅
[Node 3] 정책 위반 탐지                ← Agent 3 ✅
[Node 4] 위협 분류 (Taxonomy)          ← Agent 1 (일부)
[Node 5] 위험도 스코어링               ← RiskScorer + 전체 에이전트
[Node 6] 의사결정 (Conditional Routing) ← Decision Maker
[Node 7] 로그 & 모니터링               ← ResponseAutomation

[미할당 Node] 비정상 행동 탐지         ← Agent 4 (원래 계획에 없음)
[비표준 추가] 사용자 신뢰도 분석       ← Agent 2 (원래 계획에 없음)
```

## Agent 1: 데이터분류 (민감정보 탐지)

### ✅ 현재 상태
- **역할**: Node 2 (민감정보 분류) + 마스킹 규칙
- **구현**: Taxonomy 기반 + LLM 분류
- **출력**: data_type, sensitivity_level, masking_rules, confidence

### 🎯 Gateway 역할 충실도
**60/100** - 부분적으로 충실하나 금융권 특화 부족

#### 문제점 분석

**1) PII(Personal Identifiable Information) 탐지 약함**
- 현재: 일반적인 데이터 분류만 함
- 원래 목표: "개인신용정보, PII, 고객정보 무단 제공" 감지 (T002 위협)
- 문제: Taxonomy의 15개 카테고리가 PII를 충분히 세분화하지 않음

**2) 금융권 특화 카테고리 부족**
- 현재: 주민번호, 계좌번호, 신용카드번호 기본 패턴만
- 필요한 것:
  - 거래정보 (거래금액, 거래일시, 상대방)
  - 신용도 정보 (신용등급, 신용점수)
  - 고객 자산정보 (보유자산, 대출정보)
  - 거래상담 내용 (개인 상담 기록)

**3) 극비/대외비 마킹이 불명확**
- 현재: sensitivity_level로 구분만 함
- 필요한 것: 마킹 규칙이 실제로 마스킹될 때 어떻게 적용되는지 추적

#### 💡 개선 제안 (확인 필요)

**Q1: Taxonomy를 금융권 특화로 확장해야 할까?**
- 현재 15개 카테고리를 30개로 확대?
- PII 카테고리를 5개에서 15개로?
- 의견: 어느 정도까지 세분화해야 한다고 생각하시나요?

**Q2: Agent 1의 마스킹 규칙이 실제로 적용되는가?**
- 현재: masking_rules를 생성만 하고 실제 적용은?
- 의견: ResponseAutomation에서 마스킹을 제대로 구현했나요?

## Agent 2: 컨텍스트분석 (사용자 신뢰도)

### ⚠️ 현재 상태
- **역할**: 사용자 신뢰도 계산 + 정당성 판정
- **구현**: 부서/직급/시간/접근패턴 기반 가중점수
- **출력**: user_score (0-100), legitimacy, risk_assessment

### 🎯 Gateway 역할 충실도
**40/100** - 원래 목표에 없는 기능, 필요성 불명확

#### 문제점 분석

**1) 프로젝트 정의에 없는 기능**
- 원래 목표: 7개 Node에 "사용자 신뢰도" 분석이 없음
- 현재: 왜 이 기능을 추가했는지 불명확
- 문제: 과도한 설계 가능성

**2) Gateway의 핵심과 맞지 않음**
- Gateway의 목표: "**요청 내용**을 분석하여 위협 탐지"
- 현재: "**사용자**를 분석하여 신뢰도 평가"
- 예시: 재무팀 직원과 영업팀 직원이 같은 "고객정보 요청"을 했을 때
  - Agent 1: "고객정보" = 기밀 데이터
  - Agent 2: 재무팀은 신뢰도 높음, 영업팀은 낮음
  - 문제: 영업팀이 실제로 고객정보에 접근 권한이 있어도 "낮은 신뢰도"로 차단될 수 있음

**3) RiskScorer에서 가중치 불명확**
- Agent 2의 user_score가 최종 risk_score에 어떻게 반영되는가?
- 현재: risk_scorer.py에서 context_result를 어떻게 사용하는지 확인 필요

#### 💡 개선 제안 (확인 필요)

**Q3: Agent 2를 정말 필요한가?**
- 옵션 A: 삭제하고 Agent 3 (정책검증)에 권한 확인 통합
- 옵션 B: 역할을 변경해서 "요청 컨텍스트 분석"으로 전환 (부서 맥락, 업무 목적 등)
- 옵션 C: 유지하되, RiskScorer에서 가중치를 명확히 (예: 낮은 가중치 10%)
- 의견: 어느 방향이 맞다고 생각하시나요?

**Q4: 사용자 신뢰도와 정책 검증의 경계는?**
- Agent 2: "이 사용자는 의심스러운가?"
- Agent 3: "이 사용자가 이 데이터에 접근할 권한이 있는가?"
- 둘의 관계를 명확히 해야 하지 않을까요?

## Agent 3: 정책검증 (규제/정책 위반 탐지)

### ✅ 현재 상태
- **역할**: Node 3 (정책 위반 탐지)
- **구현**: 금융감독 규제 + 부서별 접근 권한
- **출력**: policy_violation, violation_type, approval_required

### 🎯 Gateway 역할 충실도
**70/100** - 기본 기능은 있으나 커버리지 부족

#### 문제점 분석

**1) "데이터 타입"에 대한 정책만 확인**
- 현재: "계좌번호"를 누가 접근할 수 있는가만 확인
- 필요한 것: "극비 프로젝트 정보 공개" 같은 **요청 내용** 기반 정책 위반 탐지
- 예시:
  ```
  입력: "2026년 상반기 신사업 전략 자료 공개해줘"
  Agent 1: "신사업 전략" = 극비 데이터
  Agent 3: ??? (현재는 "극비"라는 카테고리 없음)
  ```

**2) Regulatory Rules가 하드코딩됨**
- 현재: agent_3_policy.py에 정책이 직접 코딩되어 있음
- 실제 금융감독 정책은 YAML 파일에서 로드해야 함
- 문제: 정책 변경 시 코드 수정 필요

**3) 규제별 세부 기준이 부족**
- 현재: 데이터 타입만으로 판단
- 필요한 것:
  - 거래금액 기준 (예: 1억 이상 = 승인 필요)
  - 시간대 기준 (예: 야간 요청 = 추가 검증)
  - 빈도 기준 (예: 대량 요청 = 차단)

#### 💡 개선 제안 (확인 필요)

**Q5: Agent 3이 "요청 내용" 기반 정책도 확인해야 하는가?**
- 현재: 데이터 타입 + 접근 권한만
- 제안: 요청의 의도를 분석해서 정책 위반 탐지
- 예: "극비 프로젝트 정보 공개" 요청은 명시적으로 정책 위반
- 의견: 이를 Agent 3에 추가할까, 아니면 다른 Agent에서?

**Q6: 정책을 YAML에서 동적으로 로드할 필요가 있나?**
- 현재: 하드코딩됨
- 장점: 빠른 개발
- 단점: 정책 변경 시 코드 수정 필요

## Agent 4: 이상행위탐지 (비정상 접근 패턴)

### ⚠️ 현재 상태
- **역할**: 사용자의 비정상적 접근 패턴 탐지
- **구현**: 시간/빈도/IP/기기 이상 탐지
- **출력**: anomaly_score (0-100), anomaly_type, risk_level

### 🎯 Gateway 역할 충실도
**50/100** - 흥미로운 기능이지만 원래 목표와 분리됨

#### 문제점 분석

**1) 프로젝트 정의에 없음**
- 원래 목표: 7개 Node에 "이상 행위 탐지"가 없음
- 원래 계획: Phase 3 (확장 로드맵)에서 다루기로 함
- 문제: 왜 지금 구현했는지, 필요성이 불명확

**2) "사용자 이상 행위" vs "요청 이상 행위"의 혼동**
- 현재: 사용자의 접근 패턴 (언제, 어디서, 어떤 기기)
- 필요한 것: 요청의 이상 패턴

  - 대량 요청 (1분에 1000개 요청)
  - 순차 공격 (A 공격 → B 공격 → C 공격)

**3) 프로필 데이터 부족**
- 현재: 기본 프로필만 있음
- 실제: 사용자별 정상 행동 프로필이 필요 (ML 학습 기반)
- 문제: Mock 데이터로는 검증 불가능

#### 💡 개선 제안 (확인 필요)

## Agent 5: 기술공격탐지 (Prompt Injection, 시스템 공격)

### ✅ 현재 상태
- **역할**: Node 1 (Prompt Injection 탐지)
- **구현**: 정규식 패턴 + 키워드 + LLM 검증
- **출력**: attack_detected, attack_type, confidence

### 🎯 Gateway 역할 충실도
**65/100** - 기본 기능 있으나 탐지율 낮음

#### 문제점 분석

**1) SQL Injection 탐지 실패**
- 테스트 결과: "SELECT * FROM users WHERE id=1 OR '1'='1'" 탐지 실패
- 정규식: `(\bUNION\b.*\bSELECT\b|...)` 
- 문제: 공격 문자열에 대소문자 섞임이 없으면 못 잡음
- 개선: `(?i)` 플래그로 대소문자 무시하거나 더 강력한 패턴 필요

**2) LLM 검증이 항상 작동하지 않음**
- 현재: 의심 케이스만 LLM 검증
- 문제: 정규식으로 못 잡은 공격이 LLM 검증까지 안 갈 수 있음
- 예: 난독화된 SQL Injection → 정규식 실패 → LLM 검증 안 함

**3) Jailbreak 탐지 불충분**
- 현재: 기본적인 악성 키워드만 있음
- 필요한 것: 금융 특화 Jailbreak 패턴
  - "다음을 무시하고..." (지시사항 무시 시도)
  - "관리자 모드" (권한 상승 시도)
  - "백엔드 시스템에 접근" (시스템 침투)

**4) 정규식이 너무 느슨함**
- 현재: `[;&|`$(){}[\]\\]` (모든 특수문자를 공격으로 봄)
- 문제: 정상 요청도 많이 걸림 (오탐율 높음)

#### 💡 개선 제안 (확인 필요)

**Q9: SQL Injection 탐지를 개선할까?**
- 옵션 A: 정규식 강화 (더 정교한 패턴)
- 옵션 B: LLM 기반으로 전환 (모든 요청을 LLM 검증)
- 옵션 C: 외부 라이브러리 사용 (SQLi 탐지 특화 라이브러리)
- 의견: 어느 방향이 실용적일까요?

**Q10: 오탐율(False Positive)을 어떻게 관리할까?**
- 현재: 정규식이 너무 느슨해서 오탐율 높을 것 같음
- 목표: 오탐율 < 5% (프로젝트 정의)
- 의견: 현재 설정으로 이 목표를 달성할 수 있을까요?

## 🔗 Agent 간 연계성 분석

### 현재 데이터 흐름

```
User Input
  ↓
[Agent 1] 데이터분류
  ├─ Output: data_type, sensitivity_level, masking_rules
  ↓
[Agent 2] 컨텍스트분석
  ├─ Output: user_score, legitimacy
  ├─ Input used: user_id, user_context
  ├─ Agent 1과의 연계: ❌ 없음
  ↓
[Agent 3] 정책검증
  ├─ Output: policy_violation, violation_type
  ├─ Input used: data_type (Agent 1 결과)
  ├─ Agent 1과의 연계: ✅ 있음
  ├─ Agent 2와의 연계: ❌ 없음
  ↓
[Agent 4] 이상행위탐지
  ├─ Output: anomaly_score, anomaly_type
  ├─ Input used: user_context
  ├─ 다른 에이전트와의 연계: ❌ 없음
  ↓
[Agent 5] 기술공격탐지
  ├─ Output: attack_detected, attack_type
  ├─ Input used: user_input
  ├─ 다른 에이전트와의 연계: ❌ 없음
  ↓
[RiskScorer]
  ├─ Input: 모든 에이전트 출력
  ├─ Output: risk_score (0-100)
  ├─ 가중치:
  │  - Agent 1 (data_sensitivity): 40%
  │  - Agent 2 (user_score): 35%
  │  - Agent 3 (policy_violation): 15%
  │  - Agent 4 (anomaly): 10%
  │  - Agent 5 (attack): 미할당 ❌
  └─ 문제: Agent 5가 가중치에 없음!
```

### 문제점

**1) Agent 간 의존성이 약함**
- Agent 1 → Agent 3 (✅)
- 나머지는 독립적으로 작동
- 문제: 에이전트가 서로의 결과를 활용하지 않음

**2) Agent 5가 RiskScorer에 반영 안 됨**
- Agent 5 (공격 탐지): attack_detected, confidence
- RiskScorer: 이를 어떻게 점수로 변환할까?
- 현재: 명시적 가중치 없음

**3) 우선순위 부정확**
- 현재 가중치: 데이터민감도(40%) > 사용자신뢰도(35%) > 정책(15%) > 이상행위(10%)
- 문제: 공격 탐지(Agent 5)가 가장 중요하지 않나?
- 예: SQL Injection 공격(Agent 5)이 탐지되면 위험도는 100이어야 하는데, 가중치에 없음
